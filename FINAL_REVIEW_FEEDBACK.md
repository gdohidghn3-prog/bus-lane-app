## 1. [P1] alerts 휴일 누락 범위가 빠져 있음

- 문서 위치: `FINAL_REVIEW.md:31-35`
- 실제 코드 위치: `apps/api/src/routes/alerts.ts:73-76`

### 설명

문서에서는 휴일 컨텍스트 누락 대상을 목록 API와 상세 API로만 적고 있습니다.  
하지만 실제로는 `alerts` 엔드포인트도 `evaluateRules()` 호출 시 `context.isHoliday`를 넘기지 않고 있습니다.

이 범위가 문서에서 빠지면, 수정 요청을 반영하더라도 핵심 알림 경로가 그대로 오동작할 수 있습니다.

### 전달 문구

`휴일 컨텍스트 누락 이슈는 목록/상세 API뿐 아니라 alerts 엔드포인트에도 동일하게 존재하므로 수정 범위에 반드시 포함해야 합니다.`

## 2. [P2] null time 기본값 보정 제안은 위험함

- 문서 위치: `FINAL_REVIEW.md:60-74`
- 실제 스키마 위치: `apps/api/seed/001_schema.sql:30-31`

### 설명

문서에서는 `start_time` 또는 `end_time`이 `null`일 경우 `00:00` / `23:59`로 보정하자는 방향을 제안하고 있습니다.

하지만 현재 스키마에서는 두 컬럼 모두 `NOT NULL`입니다.  
이 상황에서 조용히 기본값으로 보정하면, 잘못된 데이터가 들어왔을 때 전일 규제로 오해되어 더 큰 오동작을 만들 수 있습니다.

따라서 이 항목은 "기본값 보정"보다 아래 방향이 더 안전합니다.

- 데이터 검증 실패 처리
- 로그 기록 후 스킵
- 운영 데이터 정합성 점검

### 전달 문구

`null time 대응은 기본값 보정보다 데이터 오류로 처리하는 방향이 안전합니다. 현재 스키마가 NOT NULL이므로, 이런 값이 들어오면 조용히 보정하지 말고 검증 실패나 로그/스킵으로 다루는 것이 맞습니다.`

## 3. [P2] N+1 query 표현은 사실과 다름

- 문서 위치: `FINAL_REVIEW.md:140-145`
- 실제 코드 위치: `apps/api/src/services/segmentService.ts:46-55`

### 설명

문서에서는 `getSegmentsInBBox()`를 N+1 쿼리 패턴이라고 적고 있습니다.

하지만 실제 구현은 아래 구조입니다.

- 세그먼트 목록 조회 1회
- 규칙 목록 조회 1회
- 이후 메모리에서 매핑 및 필터링

즉, 전형적인 N+1 DB 쿼리 문제는 아닙니다.  
정확한 문제는 "공간 인덱싱 없이 넓게 읽어온 뒤 애플리케이션에서 후처리한다"는 점입니다.

### 전달 문구

`이 부분은 N+1 query라기보다, 공간 조건을 DB에서 충분히 소화하지 못하고 애플리케이션 레벨에서 후처리 비용이 커지는 구조로 보는 것이 더 정확합니다.`

## 4. [P2] alerts radiusMeters 프리필터 누락 이슈를 추가해야 함

- 실제 코드 위치: `apps/api/src/routes/alerts.ts:30-46`
- 상태: `FINAL_REVIEW.md`에 아직 미반영

### 설명

알림 API는 요청값으로 `radiusMeters`를 받습니다.  
그런데 실제 사전 bbox 필터는 고정값 `delta = 0.006`을 사용하고 있습니다.

즉, 사용자가 더 큰 반경을 요청해도 시작점이 그 고정 박스 밖에 있는 세그먼트는 거리 계산 전에 제외됩니다.  
결과적으로 실제 반경 내 세그먼트가 누락되는 false negative가 발생할 수 있습니다.

### 전달 문구

`alerts API는 radiusMeters를 받지만, 사전 bbox 필터가 고정 delta 기반이라 더 큰 반경 요청 시 실제 대상 세그먼트를 거리 계산 전에 놓칠 수 있습니다. 이 이슈는 최종 리뷰 문서에 별도 항목으로 추가하는 것이 좋습니다.`
